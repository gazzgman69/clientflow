Phase B — Minimal hardening (only what Phase A flags)

Implement only targeted fixes; keep them small and reversible:

B1. Fail-fast tenant guards (storage)

Any create/update that can write without tenantId → add a guard: throw with structured log.

Any read without tenant filter → add tenant scoping.

B2. Enforce schema constraints (flags)

Prepare (don’t execute unless told) migrations:

ALTER TABLE <table> ADD CONSTRAINT <tbl>_tenant_not_null CHECK (tenant_id IS NOT NULL);

Add tenant-scoped unique indexes, e.g.
CREATE UNIQUE INDEX IF NOT EXISTS uq_events_tenant_ext ON events(tenant_id, external_event_id);

Only auto-apply if TENANT_SCHEMA_ENFORCE=1.

B3. RLS verification hook

In DB init or request middleware, set app.tenant_id per request, and add a self-check that queries pg_policies for required tables; log if missing.

B4. Jobs & webhooks

Ensure workers/webhooks wrap processing with tenant context. If missing, add a lightweight wrapper (no refactor) that sets tenant for the duration of the handler.

B5. OAuth callback hardening

Verify/require tenantId from signed state; remove any fallback to null/default.

Upsert integrations on (tenant_id, provider, provider_account_id); deactivate duplicates; never insert tenant_id = NULL.

B6. UI/API safety

Remove any leftover user/tenant hardcodes; require session + tenant on fetchers.

Add a tiny log in /api/* list routes stating branch + tenantId + count.

B7. CI guard

Add a lightweight CI grep/ESLint block to prevent re-introducing tenantId-less queries or 'test-user' in client/src & server/src.
