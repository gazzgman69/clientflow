Goal: Prove and enforce full multi-tenant isolation across API, storage, jobs, OAuth, UI, and data. Then confirm parity between Dashboard and Calendar views per tenant, and validate that the SUPERADMIN impersonation system works securely.

Rules

Phase A = read-only.

Any seeding must be idempotent, behind TENANT_TEST_SEED=1.

Keep changes minimal & additive; do not refactor unrelated features.

Respect RLS/withTenant() everywhere.

For impersonation: only test via the existing SUPERADMIN impersonation routes (/api/admin/impersonate/start, /api/admin/impersonate/end, /api/admin/impersonate/status).

Phase A — Read-only Audit

Show where tenantId is resolved in middleware and passed to storage.

Confirm OAuth callbacks + integrations use tenantId, not null/default.

Verify schema constraints: tenant_id NOT NULL + RLS policies.

Check background jobs/webhooks enumerate tenants correctly.

Confirm UI/API fetchers (Dashboard + Calendar) do not hardcode tenants.

Verify contacts/projects/events use Postgres not MemStorage.

Check impersonation system:

SUPERADMIN cannot impersonate another SUPERADMIN

Audit logs exist for start/end

Tenant context switches properly

Phase B — Minimal Hardening (only if flagged)

Add fail-fast guards in storage: reject writes with missing tenantId.

Prepare migrations:

ALTER TABLE <table> ADD CONSTRAINT <table>_tenant_not_null CHECK (tenant_id IS NOT NULL);
CREATE UNIQUE INDEX IF NOT EXISTS uq_events_tenant_ext ON events(tenant_id, external_event_id);


Only auto-apply if TENANT_SCHEMA_ENFORCE=1.

Add log self-checks for RLS presence at startup.

Fix any OAuth callbacks inserting with tenant_id=NULL.

Add CI guard against tenantId="default" or "test-user" hardcodes.

Phase C — Switch-Tenant Test (end-to-end)

Tenant A session: auth, hit Dashboard + Calendar, record 30-day event counts.

Tenant B session: repeat, record counts.

Cross-check: A cannot see B’s events, B cannot see A’s, and Dashboard = Calendar parity inside each tenant.

Run SQL probes to confirm no orphaned records:

SELECT tenant_id, COUNT(*) FROM events GROUP BY tenant_id;
SELECT COUNT(*) FROM events WHERE tenant_id IS NULL;

Phase D — SUPERADMIN Impersonation Validation

As SUPERADMIN, call:

POST /api/admin/impersonate/start { userId: <tenantA_user>, tenantId: tenantA }
GET /api/events  // should return ONLY tenantA’s events
POST /api/admin/impersonate/end


Repeat for tenantB.

Confirm:

Events returned belong only to impersonated tenant

Audit log entries created for start/end

Session regenerates correctly

Attempting to impersonate another SUPERADMIN is blocked

Nested impersonation is blocked

Phase E — Acceptance

Pass if all true:

Tenant-scoped storage + RLS confirmed.

Dashboard/Calendar parity per tenant.

No orphaned rows.

SUPERADMIN impersonation works, logs actions, enforces protections.

Fail if:

Any cross-tenant leakage.

Any tenant_id=NULL records.

Impersonation bypasses protections.

Output

Provide:

File:line evidence for tenant resolution, RLS, and impersonation guards.

Seed summary if TENANT_TEST_SEED=1.

Dashboard + Calendar API counts per tenant.

Impersonation test results (success/failure, logs).
