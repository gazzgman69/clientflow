Constraints:

1) Storage interface + implementations

Add these methods to the storage interface and implement in DrizzleStorage and MemStorage. All require tenantId and must be tenant-scoped:

// Read all saved configs for the tenant
getEmailProviderConfigs(tenantId: string): Promise<EmailProviderConfig[]>;

// Insert or update a provider config (upsert by (tenantId, providerCode))
upsertEmailProviderConfig(input: {
  tenantId: string;
  providerCode: string;       // e.g., 'gmail','microsoft','icloud','yahoo','custom','sendgrid',...
  authMode: 'oauth' | 'appPassword' | 'apiKey';
  isActive: boolean;
  config: Record<string, unknown>; // encrypted at rest where sensitive
}): Promise<EmailProviderConfig>;

// Delete provider config
deleteEmailProviderConfig(tenantId: string, providerCode: string): Promise<void>;

// Optional: lightweight credential check
verifyEmailProviderConfig(tenantId: string, providerCode: string): Promise<{ ok: boolean; message?: string }>;

Schema (migrate if table is missing)

Create or confirm a table (discover naming, don’t duplicate) like email_provider_configs with:

id uuid pk

tenant_id (FK → tenants.id) NOT NULL

provider_code text NOT NULL

auth_mode text NOT NULL

is_active boolean default false

config_json jsonb NOT NULL (store non-secret bits here)

created_at, updated_at timestamps

Unique (tenant_id, provider_code)

Secrets (app passwords, API keys, OAuth refresh tokens) must use the existing secure/encrypted store (e.g., secureStore/KMS) referenced from config_json by key name only.

2) API routes

Add/confirm REST endpoints (under the existing email router; don’t create a new top-level router):

GET /api/email/providers

middleware: ensureUserAuth, tenantResolver, requireTenant

returns array from getEmailProviderConfigs(tenantId)

PUT /api/email/providers/:providerCode

body: { authMode, isActive, config } (non-secret fields only)

upserts via upsertEmailProviderConfig

any secrets in the payload must be written via secure store and not echoed back

DELETE /api/email/providers/:providerCode

calls deleteEmailProviderConfig

POST /api/email/providers/:providerCode/verify

calls verifyEmailProviderConfig (uses the adapter to check connectivity minimally)

respond { ok, message }

All endpoints must read tenantId from the resolved request context and never accept it from the client.
